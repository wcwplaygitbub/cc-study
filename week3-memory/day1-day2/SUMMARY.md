## 一、当天问题记录

### Q1: MEMORY.md 和普通 .md 记忆文件有什么区别？为什么这样设计？

**A:** MEMORY.md 是**索引文件**，只有一行一条的引用；真正的记忆内容存在各类型的子目录里（user/feedback/project/reference）。这样设计是因为 MEMORY.md 每次对话都会加载进 system prompt，必须保持小而精简；具体内容可以很大，按需用工具读取。

### Q2: cc-haha 把记忆分成哪 4 类？各自的用途是什么？

**A:** user（用户信息）、feedback（纠正/确认）、project（项目进展）、reference（外部系统指针）。user 帮 Agent 理解用户是谁；feedback 避免重复犯错；project 记录上下文和决策；reference 记住去哪儿找信息。

### Q3: 为什么"代码模式、Git 历史、项目结构"不该存记忆？

**A:** 因为这些可以从当前项目状态推导。记忆只存"不可推导"的信息——用户偏好、项目外的事实、外部系统状态。如果从代码能直接看到架构，不需要重复存一份到记忆里。

### Q4: feedback 记忆的结构是什么？为什么要有 Why 和 How to apply？

**A:** 规则 + **Why** + **How to apply**。Why 让 Agent 理解背后的原因，从而在边缘情况下能正确判断；How to apply 告诉 Agent 在哪里适用。两者结合让 feedback 在新场景下也能正确应用，而不只是死记硬背。

### Q5: 记忆的写流程是什么？为什么分成两步？

**A:** 1）写单独的 .md 文件到对应类型目录；2）更新 MEMORY.md 加一行索引。两步是因为 MEMORY.md 是索引不能存内容，内容放单独文件按需读取，索引保持精简。

## 二、核心理解

### 文件化记忆的价值

人、Agent、Git 都能理解和操作同一份记忆文件，不需要专门的工具或界面。

### 4 类记忆的分类边界

feedback 里的 scope 标注（private vs team）解决了"个人偏好"和"项目公约"的区分问题。

### 今天最大的收获
