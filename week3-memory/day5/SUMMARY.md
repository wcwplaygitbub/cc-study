## 一、当天问题记录

### Q1: cc-haha 用文件存储记忆，优点和缺点各是什么？

**A:** 优点：人机共读、Git 可追踪、无需额外基础设施、迁移简单。缺点：无结构化查询、并发写入需协调、大规模检索不如向量数据库。

### Q2: cc-haha 的 4 类记忆各解决什么问题？为什么需要分类？

**A:** user 解决"用户是谁"；feedback 解决"避免重复犯错"；project 解决"上下文和决策"；reference 解决"外部系统指针"。分类让 Agent 知道什么时候该写、写成什么、怎么用，避免记忆类型混乱。

### Q3: cc-haha 为什么用 MEMORY.md 作为索引？

**A:** 索引文件小（200行/25KB 上限），每次加载 cost 可控；内容文件大但按需读取；避免每次加载所有记忆导致 token 爆炸。

### Q4: OpenHuman 的 auto-fetch 和 cc-haha 的主动写入，哪个更适合你？

**A:** 取决于场景：代码工具场景（cc-haha）更适合主动写入——记忆围绕项目，信息量可控，需要精确；个人助手场景（OpenHuman）更适合 auto-fetch——信息量大、来源多，需要自动化。老大用的是 cc-haha 做代码工具，主动写入更合适。

### Q5: 记忆系统的 3 个局限性是什么？怎么缓解？

**A:** 1）记忆过载 → 截断机制 + 分类；2）记忆一致性 → Agent 主动验证 + MEMORY_DRIFT_CAVEAT；3）并发冲突 → 单 Agent 操作限制 + Daily Log append-only 模式。

## 二、Week 3 整体总结

### 记忆系统的本质

cc-haha 的记忆是"文件化 + 分类索引"，不是什么黑箱向量数据库，就是一堆 Markdown 文件 + 一个索引 + Grep 搜索。简单但有效。

### cc-haha vs OpenHuman

OpenHuman 是"自动化的知识库"，适合个人 AI 助手场景；cc-haha 是"人机协作的记忆"，适合代码工具场景。两者设计哲学不同。
